# v1.x 可维护性架构冻结

> **生效版本：** v1.10.2  
> **生效日期：** 2026-07-14  
> **状态：** 正式冻结  
> **执行依据：** `docs/superpowers/specs/06-one-shot-final-closure-spec.md`

## 1. 背景

ShineHeKnowledge v1.10.2 完成一次性最终收尾（Spec 06）后，v1.x 线的可维护性专项、迁移治理和发布一致性全部收尾完成。本文档正式冻结 v1.x 架构治理边界：

- v1.x 不再为"更纯粹""更整洁""更少行数"发起重构；
- 剩余兼容层与架构优化统一登记为 v2.0 技术债，**不再以"当前项目未完成"为由继续重构**；
- v1.x 后续仅接受定向修复，产品功能开发回归主线。

本冻结不是"禁止修改"，而是"禁止以架构美化为目的的修改"。

## 2. v1.x 只允许处理

以下类型的变更可在 v1.x 线继续进行，无需重新打开可维护性专项：

- **功能 Bug** — 实际行为与公开契约不符的缺陷；
- **安全问题** — 漏洞、注入、越权、敏感信息泄露等；
- **数据损坏风险** — 迁移、写入、并发导致的丢数据或脏数据；
- **性能严重回退** — 关键路径（检索/问答/索引）可量化的显著退化；
- **CI 回归** — 已绿门禁变红、门禁被绕过、新增已知回归未被阻止；
- **用户升级阻塞** — 从旧版本升级到 v1.10.x 时无法启动或数据不可用；
- **明确的公开契约错误** — MCP 工具名/参数/返回结构、REST 契约、错误码的回归。

此类修复应附带最小化测试，且不得顺手扩大到第 3 节所列范围。

## 3. v1.x 不再处理

以下动机在 v1.x 线**不再受理**，应登记到第 4 节并转入 v2.0：

- 为了"更纯粹"而继续拆分模块；
- 为了减少文件行数而重构；
- 兼容层提前删除（`Database._instance`、`get_active_container`、`src/mcp_server.py` 等）；
- Provider 重新归属；
- `SearchService` 进一步变薄；
- MCP 直接 SQL 的全量下沉；
- 非必要的 Repository 重构；
- 没有用户影响的架构美化。

发现上述情形时，只登记，不在 v1.x 修改。

## 4. v2.0 技术债清单

以下项目统一登记但不执行。它们**不得再被描述为 v1.10.2 "未完成"**：

| # | 技术债 | 当前状态 | 处置 |
|---|--------|----------|------|
| 1 | `Database._instance` 兼容入口 | 保留，仓库内引用受 debt gate 监控 | v2.0 评估移除 |
| 2 | `get_active_container()` 兼容入口 | 保留，白名单外引用为 0 | v2.0 评估移除 |
| 3 | `execute_primary_legacy` shim | 保留 | v2.0 评估移除 |
| 4 | legacy MCP aliases | 仅 `legacy` 配置档启用 | v2.0 评估移除 |
| 5 | `src/mcp_server.py` 兼容层 re-export | 保留，作为 monkeypatch surface | v2.0 评估内联 |
| 6 | `SearchService` 私有 retrieval helpers | 保留 | v2.0 评估下沉 |
| 7 | MCP 残余直接 SQL | 保留 | v2.0 评估下沉到 Repository |
| 8 | Wiki Projection Provider 归属 | 保留，serving 入口为 `WikiServingGate` | v2.0 评估重归属 |
| 9 | MCP 大型域文件进一步拆分 | 保留 | v2.0 评估拆分 |

这些项目的存在不构成 v1.x 竣工的阻碍。`tools/report_closure_debt.py --strict` 持续监控其引用边界，防止失控扩散。

## 5. 重新打开专项的条件

只有出现以下情况，才允许重新打开 v1.x 可维护性专项：

- 生产数据损坏；
- 安全漏洞；
- 公共契约破坏；
- 迁移失败；
- CI 无法阻止已知回归；
- Retrieval / Wiki 关键指标实际下降。

任意一项成立时，可发起定向专项；否则 v1.x 保持冻结，相关工作进入 v2.0 规划。

## 6. 结论

> v1.x 可维护性、迁移治理和发布一致性全部收尾完成。  
> 后续工作默认回归产品功能开发。任何进一步架构优化必须进入 v2.0 规划，不再作为 v1.x 竣工条件。
