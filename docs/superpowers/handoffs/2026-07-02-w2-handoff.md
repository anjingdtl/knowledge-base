## 5. 关键上下文与坑（必读，避免踩 W1 已填的坑）

### 5.1 Gap A 决策 = A2 方案（已定稿）
`entities`/`concepts` 页 frontmatter **只有 `knowledge_id`，无 `source_ids`/`key_entities`**（第一阶段 `wiki_entity_updater` 没写）。W2 的 `WikiParentRetriever` 要用 **`knowledge_id` 回查 source 页摘要**（A2 方案），**不动已交付的第一阶段编译器**。`syntheses`/`comparisons` 页带 `source_ids`（列表），直接回查多 source。`sources` 页自身即 source，跳过。

### 5.2 scale 在 WikiReadStage 算（不是 AgenticRouter）— W1 修正的时序 bug
W1 plan 原写「scale 在 `AgenticRouter.route()` 内算」，但 `WikiReadStage` 在调 agentic 的 `VectorSearchStage` **之前**执行，`ctx.metadata["scale"]` 那时还不存在。最终 scale 在 `WikiReadStage` 算并缓存。**W2 若涉及管线挂载，务必先画 stage 执行顺序图确认时序。** 详见 `PROGRESS.md` W1 段「设计要点」。

### 5.3 wiki 候选 schema 已统一（W1 Task 1.0 定）
```python
{
  "id": "wiki:<page_type>:<slug>",           # 与检索候选 page_id:block_id 不冲突
  "text": <body>,
  "metadata": {"page_type", "title", "path", "knowledge_id",
               "key_entities", "source_hash", "wiki_hit_score"},
  "match_channels": ["wiki_read"],
}
```
W2 的 `parent_context` 字段对齐 block parent-child（`hybrid_search.enrich_with_parent_context`），加到候选顶层或 metadata，保持与 `CitationBuilder` 一致。

### 5.4 Gap B（W4 前置，W2 不涉及）
文件系统 wiki 缺测量基础设施（`shinehe wiki lint` 查旧 SQLite `wiki_pages` 表，对 `wiki/*.md` 无效）。W4 eval 扩展前要补文件系统 wiki lint/统计工具。

### 5.5 Config.get 支持任意深度点号路径
`Config.get("rag.size_aware.xxx", default)` 缺段返回 default，无需改 `config.py`。但注意 `project_setup._wiki_first_defaults()` 经 `config.update()` **浅合并**——新配置段要放进 `_build_local_config`/`_build_provider_config` 各自的 dict，不能放进 `_wiki_first_defaults`（会整体覆盖 `rag` 段，W1 已踩过）。
