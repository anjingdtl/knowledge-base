# v1.9.x → v1.10.0 Maintainability Closure

## Summary

v1.10.0 completes Spec 04 (maintainability closure):

- Retrieval is **unified-only** (`RetrievalOrchestrator` + Policy + `RawRetriever` + `VerifiedFusion`)
- Answer assembly lives in `src/answering/`
- MCP `server.py` is a thin registration shell; tools live in `src/mcp/tools/*`
- Container capability groups are real Providers (construction + lifecycle)
- Alembic head gate blocks write-mode boot when schema is behind head
- Legacy retrieval dual-path (`legacy` / `shadow` orchestrator pipelines) removed

## Config changes

| Key | Before | After |
|-----|--------|-------|
| `retrieval.orchestrator` | `legacy` \| `shadow` \| `unified` (default unified from 1.9.1) | **unified only**; legacy/shadow values warn and map to unified |
| `answer.orchestrator` | legacy/shadow aliases ignored | unified only |

## Operator notes

1. **No schema rewrite required** for this release if already on Alembic head.
2. Run `alembic upgrade head` if migration gate refuses write boot.
3. Optional: set `storage.migration_gate.enabled: true` (default) and `storage.readonly: true` for diagnostic boot when behind head.
4. Remove any automation that set `retrieval.orchestrator: legacy` for rollback; that path no longer exists.

## Rollback

Code rollback to a prior tag (e.g. v1.9.0) is required if you need the dual-path orchestrator. Config alone cannot restore legacy retrieval after v1.10.0.
