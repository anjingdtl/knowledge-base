# ShineHeKnowledge v1.10.1 Release Notes

**Final Migration Governance Closure** — Alembic is now the sole runtime schema authority.

v1.10.1 completes Spec 05 (final migration governance). It is a governance patch on top of v1.10.0: **no public API/MCP contract changes, no retrieval/Wiki semantics changes, no new database revision.**

## Highlights

- **Alembic is the sole runtime schema authority** — new/empty databases are created exclusively via `alembic upgrade head`; the runtime no longer executes `_SCHEMA` or calls `_migrate()` (WP3, WP5)
- **Migration Gate precedes Database open** — `inspect_database_bootstrap()` inspects on a read-only SQLite connection and enforces the gate before any `Database.open_runtime()`; behind-head write boot is refused (WP2)
- **Unstamped legacy DB safe migration** — `allow_unstamped` defaults to `false`; the `shinehe db {status,backup,migrate,stamp,verify}` CLI uses the SQLite Backup API, detector-gated stamp, and automatic backup restore on failure (WP4)
- **Strict CI architecture gate** — `report_closure_debt --strict` exits non-zero on any residual debt (WP1)
- **Acceptance-blocking defect fixes** — cleared the ruff/mypy regression accumulated during WP2–WP5, including two real `NameError`-class defects from MCP tool extraction:
  - `preview_operation(op="ingest_file")` referenced `ingest_file` without importing it → now lazy-imported
  - `kb_capabilities.hidden_by_policy` always returned `[]` because a `globals()` guard could never see the undefined `_HIDDEN_BY_POLICY`

## What did NOT change

- Public Search / Ask / MCP contracts — unchanged (contract suite green)
- Retrieval ranking / recall / RRF / rerank — unchanged (Retrieval & Hybrid eval green)
- Wiki serving gate / Claim / Evidence semantics — unchanged
- Database file format — already at Alembic head since v1.10.0; no new revision in this release

## Upgrade

Already on v1.10.0 (Alembic head)? **No schema action required.**

```bash
pip install -e ".[all]"
shinehe db status     # confirm at head, write allowed
# Only if behind head or unstamped:
shinehe db backup
shinehe db migrate
```

## Rollback

Code rollback to v1.10.0 is safe — v1.10.0 may re-run `_SCHEMA + _migrate()` (warned in its docs). Databases already at head stay forward-compatible; no automatic downgrade is performed. Back up before rolling back code.

See `docs/migration/v1.10-to-v1.10.1-migration-governance.md` and `docs/architecture/database-migration-policy.md`.
