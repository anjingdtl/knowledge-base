# ShineHeKnowledge v1.8.0 — Verified Hybrid Convergence Correction

## Highlights

- Strict serving requires current revision validation, approved review, explicit publish, and resolvable evidence.
- Source events, durable jobs, reviews, dead letters, schedules, and health history share one Maintenance control plane.
- Search/Ask fuse Raw evidence with eligible Claims and disclose conflicts; no eligible Claim means Raw fallback.
- CI now requires Ruff, mypy, Python 3.10–3.12 tests, Docker build/health smoke, frontend build, and Windows MCP smoke.
- New backup-first configuration migration normalizes legacy modes only when requested.

## Evidence boundaries

The release eval is deterministic and executes the actual local SearchService/VerifiedAnswerService chain on a controlled 60-case fixture. It proves the gate/fusion contract, not remote embedding or LLM quality. A provider-backed evaluation must be attached separately before claiming model-quality improvement.

## Compatibility

- `wiki_first` remains runtime-compatible as `authoring`.
- `legacy` remains runtime-compatible as `evidence_only`.
- Existing MCP configurations without an explicit tool profile preserve the legacy tool surface until their owner selects one.
- The private React client keeps its independent package version (`1.4.0`); product version is `src/version.py`.

## Upgrade

Follow [v1.7 → v1.8 migration](../migration/v1.7-to-v1.8-convergence-correction.md). No migration command writes a user configuration unless `--apply` is present.
