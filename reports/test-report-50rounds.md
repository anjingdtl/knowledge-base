测试时间：2026-06-25 | 测试工具：OfficeCLI + MCP | 测试轮次：50轮

**一、测试概述**

本次测试通过 shinehe-kb 知识库的 MCP（Model Context Protocol）工具接口，对知识库的稳定性和知识召回准确性进行了全面验证。测试覆盖6种API接口（kb\_search、kb\_ask、kb\_search\_fulltext、kb\_ask\_with\_query、kb\_route\_query、kb\_structured\_query），涉及20+个业务领域。

知识库当前状态：degraded（降级模式），总文档135条有效文档，总块数4278，向量覆盖率99.6%，但标签覆盖率仅3.7%（仅5条文档有标签），P95延迟67.7秒。标签覆盖率严重偏低是导致 kb\_route\_query 路由功能100%退化的根本原因。

**二、核心指标汇总**

总测试轮次：50轮 | 成功：48轮（96.0%） | 失败：2轮（4.0%）

失败原因：kb\_ask 接口超时（MCP error -32001: Request timed out），涉及R35（供应商管理办法）和R46（单一来源采购合规指引）

知识召回准确性：总体良好，大部分查询能准确返回相关文档内容，部分 kb\_ask/kb\_ask\_with\_query 输出因文档较大被截断（>300KB）

**五、问题清单**

**【Bug-1】kb\_route\_query 路由功能100%退化（P0-严重）**

现象：全部6次 kb\_route\_query 调用均 fallback 到 hybrid search，未实现基于标签的智能路由。根因：知识库标签覆盖率仅3.7%（135条文档中仅5条有标签），路由引擎无法匹配到有效标签。影响：无法按业务分类精准检索，所有查询退化为全库混合搜索。修复建议：执行 auto\_tag 批量补标操作，将核心业务文档打上对应标签（如企微运营、管理办法、职称评审等），目标将标签覆盖率提升至80%以上。

**【Bug-2】kb\_ask 偶发超时（P1-中等）**

现象：R35（供应商管理办法）和R46（单一来源采购合规指引）两次调用均因超时失败，错误码 MCP error -32001。超时率11.1%（2/18次kb\_ask调用）。可能原因：(1) 对应文档块数较多导致LLM生成时间过长；(2) P95延迟67.7秒偏高，偶发峰值超过超时阈值。修复建议：(1) 排查上述两篇文档的块数量和大小；(2) 考虑对大文档增加分块策略或设置更长的超时时间；(3) 监控LLM API响应时间分布。

**六、结论与建议**

总体评价：shineheKB 知识库在本次50轮MCP接口测试中表现基本合格，总体成功率96.0%，知识召回准确性良好。6种API中，kb\_search、kb\_search\_fulltext、kb\_ask\_with\_query、kb\_structured\_query 四类API表现稳定（成功率100%），kb\_ask 存在偶发超时需关注，kb\_route\_query 因标签覆盖率不足导致功能退化。

核心优势：(1) 向量检索质量高，kb\_search 的 score 普遍达到0.9-1.0；(2) kb\_ask 对业务文档的理解和归纳能力强，能输出结构化回答（含表格、分章节归纳）；(3) kb\_structured\_query 标题过滤精准，能精确定位目标文档；(4) 知识库覆盖面广，120+条有效文档涵盖企微运营、各类管理办法、职称评审、渠道管理等核心业务领域。

待改进项：(1)【紧急】执行 auto\_tag 批量补标，修复 kb\_route\_query 路由退化问题；(2)【重要】排查并优化 kb\_ask 超时问题，降低P95延迟；(3)【建议】对大文档输出增加分页或摘要机制，避免截断；(4)【建议】建立知识库健康度监控看板，持续跟踪标签覆盖率、延迟等关键指标。
